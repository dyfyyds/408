# TCP协议（传输控制协议）

## TCP的基本概念

### 什么是TCP？
TCP（Transmission Control Protocol）是传输层的**面向连接的、可靠的、面向字节流**的传输层通信协议

### TCP的特点

#### 面向连接
- 通信前必须建立连接（三次握手）
- 通信结束后释放连接（四次挥手）

#### 可靠传输
- 确认和重传机制
- 序列号
- 超时重传

#### 流量控制
- 滑动窗口机制

#### 拥塞控制
- 慢启动
- 拥塞避免
- 快速重传
- 快速恢复

### TCP报文段结构

```
┌──────┬──────┬──────────┬──────────┬──────┬──────┬──────┬──────┐
│ 源端口 │ 目的端口│    序号（Sequence Number）     │ 确认号 │ 首部 │ 保留 │URG|ACK|PSH|RST|SYN|FIN│ 窗口 │ 校验和 │ 紧急指针 │
├──────┴──────┴──────────┴──────────┴──────┴──────┴──────┴──────┤
│                可选选项                                │ 填充 │
└─────────────────────────────────────────────────────────────┘
```

#### 关键字段说明

| 字段 | 大小 | 功能 |
|------|------|------|
| **源端口** | 16位 | 发送方端口号 |
| **目的端口** | 16位 | 接收方端口号 |
| **序号（Seq）** | 32位 | 本报文段所发送数据的第一个字节的序号 |
| **确认号（Ack）** | 32位 | 期望收到对方下一个报文段数据的第一个字节的序号 |
| **URG** | 1位 | 紧急指针字段有效 |
| **ACK** | 1位 | 确认号字段有效 |
| **PSH** | 1位 | 接收方尽快交付应用层 |
| **RST** | 1位 | 重置连接 |
| **SYN** | 1位 | 同步序列号（连接建立时使用） |
| **FIN** | 1位 | 发送方完成数据发送（连接释放时使用） |
| **窗口** | 16位 | 接收窗口大小 |

## TCP三次握手

### 握手过程

```
客户端                             服务端
   │                                 │
   │────────── SYN=1, Seq=x ────────►│  SYN-SENT
   │                                 │
   │◄── SYN=1, ACK=1, Seq=y, Ack=x+1─│  SYN-RCVD
   │                                 │
   │────── ACK=1, Seq=x+1, Ack=y+1 ──►│  ESTABLISHED
   │                                 │
   │◄───────────────────────────────►│  ESTABLISHED
```

### 三次握手的作用

1. **确认双方的发送和接收能力正常**
2. **指定自己的初始序列号，为可靠传输做准备**
3. **防止已失效的连接请求报文突然又传到服务器**

### 三次握手详解

#### 第一次握手（SYN=1, Seq=x）
- 客户端发送SYN包（SYN=1）
- 选择一个初始序列号x
- 进入SYN-SENT状态

#### 第二次握手（SYN=1, ACK=1, Seq=y, Ack=x+1）
- 服务器收到SYN包
- 确认序列号ACK=x+1
- 服务器也发送SYN包（SYN=1）
- 选择一个初始序列号y
- 进入SYN-RCVD状态

#### 第三次握手（ACK=1, Seq=x+1, Ack=y+1）
- 客户端收到服务器的SYN+ACK包
- 发送ACK包确认服务器的序列号
- 进入ESTABLISHED状态
- 服务器收到ACK后也进入ESTABLISHED状态

### 为什么是三次，不是两次？

- **防止已失效的连接请求报文突然又传到服务器**
- 如果只有两次，服务器会立即建立连接，浪费资源
- 三次握手确保双方都有发送和接收能力

## TCP四次挥手

### 挥手过程

```
客户端                          服务端
   │                                │
   │────── FIN=1, Seq=u ───────────►│  FIN-WAIT-1
   │                                │
   │◄── ACK=1, Seq=v, Ack=u+1 ──────│  CLOSE-WAIT
   │                                │
   │          ...（数据传输）         │
   │                                │
   │◄───── FIN=1, ACK=1, Seq=w ──────│  FIN-WAIT-2
   │              Ack=u+1            │
   │                                │
   │──── ACK=1, Seq=u+1, Ack=w+1 ──►│  TIME-WAIT
   │                                │
   │◄───────────────────────────────►│  CLOSED
   │                                │
   │──────── 2MSL等待时间 ────────────►│
   │                                │
   │───────────────────────────────►│  CLOSED
```

### 四次挥手详解

#### 第一次挥手（FIN=1, Seq=u）
- 客户端发送FIN包，发送序号为u
- 进入FIN-WAIT-1状态
- 客户端没有数据要发送了

#### 第二次挥手（ACK=1, Seq=v, Ack=u+1）
- 服务器收到FIN包，发送ACK确认
- 确认序号为u+1
- 服务器进入CLOSE-WAIT状态
- 客户端收到ACK后进入FIN-WAIT-2状态

#### 第三次挥手（FIN=1, ACK=1, Seq=w, Ack=u+1）
- 服务器发送FIN包，发送序号为w
- 服务器进入LAST-ACK状态

#### 第四次挥手（ACK=1, Seq=u+1, Ack=w+1）
- 客户端收到FIN包，发送ACK确认
- 客户端进入TIME-WAIT状态
- 等待2MSL后进入CLOSED状态
- 服务器收到ACK后立即进入CLOSED状态

### 为什么TIME-WAIT状态需要等待2MSL？

- **MSL（Maximum Segment Lifetime）**：报文最大生存时间
- 2MSL = 2 × 最大报文生存时间

#### 原因
1. **确保最后一个ACK到达服务器**
   - 如果ACK丢失，服务器会重传FIN
   - 客户端在2MSL内可以重新发送ACK

2. **防止已失效的连接请求报文出现在下一个连接中**
   - 等待网络中所有报文段消失

### 为什么挥手是四次，握手是三次？

- **服务器收到FIN后，可能还有数据要发送**
- 服务器先发送ACK确认，再发送FIN
- 所以变成四次挥手

## TCP可靠传输

### 确认机制

#### 累积确认
- ACK确认序号表示期望收到的下一个字节的序号
- 累积确认：确认号之前的所有字节都收到

#### 确认应答
```c
// 发送方发送数据
send(data, seq=x);

// 接收方确认
ack = x + len(data);
```

### 超时重传

#### 超时时间（RTO）
$$RTO = RTT_s + 4 \times RTT_d$$

- $RTT_s$：平滑往返时间
- $RTT_d$：平均偏差

#### 快速重传
- 收到3个重复的ACK
- 不需要等待超时，立即重传

### 序号机制

#### 序列号
- TCP是面向字节流的
- 每个字节都有序列号
- 序号字段的值 = 本报文段所发送数据的第一个字节的序号

## TCP流量控制

### 什么是流量控制？
**控制发送方的发送速度，使接收方来得及接收**

### 滑动窗口机制

#### 接收窗口
- 接收方根据自身缓存空间大小动态调整
- 在TCP头部的窗口字段告知发送方

#### 滑动窗口
```
发送窗口：发送方可以连续发送的数据范围

接收窗口：接收方可以接收的数据范围

窗口大小 = min(发送窗口大小, 接收窗口大小)
```

### 零窗口处理
- 接收窗口为0时，发送方停止发送数据
- 接收方定期发送零窗口探测报文

## TCP拥塞控制

### 什么是拥塞控制？
**防止过多的数据注入网络，导致网络过载**

### 拥塞控制算法

#### 1. 慢启动（Slow Start）
- 拥塞窗口cwnd从1开始
- 每收到一个ACK，cwnd += 1（指数增长）
- cwnd达到慢启动阈值ssthresh后，进入拥塞避免

#### 2. 拥塞避免（Congestion Avoidance）
- 每经过一个RTT，cwnd += 1（线性增长）
- 发生超时：ssthresh = cwnd / 2，cwnd = 1，重新慢启动

#### 3. 快速重传（Fast Retransmit）
- 收到3个重复ACK
- 立即重传丢失的报文段

#### 4. 快速恢复（Fast Recovery）
- 收到3个重复ACK后：
  - ssthresh = cwnd / 2
  - cwnd = ssthresh + 3（因为收到3个重复ACK）
  - 每收到一个重复ACK，cwnd += 1
  - 收到新数据的ACK后，cwnd = ssthresh，进入拥塞避免

### 拥塞控制的状态图

```
慢启动 → 拥塞避免 → 超时
  ↑                      ↓
  └────── 快速恢复 ───────┘
```

## TCP与UDP对比

| 对比维度 | TCP | UDP |
|---------|-----|-----|
| **连接性** | 面向连接 | 无连接 |
| **可靠性** | ✅ 可靠 | ❌ 不可靠 |
| **流量控制** | ✅ 有 | ❌ 无 |
| **拥塞控制** | ✅ 有 | ❌ 无 |
| **传输效率** | 低 | 高 |
| **数据格式** | 面向字节流 | 面向报文 |
| **适用场景** | 网页、邮件、文件传输 | 视频、语音、DNS、DHCP |

## 考试要点

### 高频考点
1. **TCP三次握手** — 过程、状态转换、为什么三次
2. **TCP四次挥手** — 过程、TIME-WAIT、为什么四次
3. **TCP可靠传输** — 确认机制、超时重传、快速重传
4. **TCP流量控制** — 滑动窗口机制
5. **TCP拥塞控制** — 慢启动、拥塞避免、快速重传、快速恢复

### 常见陷阱
- 第三次握手已经携带数据（HTTP的GET请求）
- TIME-WAIT状态持续2MSL
- 三次握手不能携带数据，第三次可以
